Inspiration

Several years ago, my girlfriend suffered from chronic fatigue syndrome and random hematomas. Giant, discolored bruises all over her body for no discernable reason.

She mentioned she had her raw genome available and that there were some tools out there you could do digging with. So over the next few months I did a deep-dive into genomics. I connected with Professor of Bioinformatics Lasse Folkersen via email over several months and we had a very productive mentorship.

Through this, one of the most useful things I was able to discover was that she had a genotype related to drug metabolism that presented singnificant risk for gastrointestinal bleeding with any NSAID (Ibuprofen, Aspirin, Naproxen, etc) consumption -- which her doctors has been prescribing her in high doses for years. We stopped the NSAID use, and the hematomas have not returned since .

Ever since then, I Have felt like the space for a personalized, accessible, and easy-to-consume personal genomics platform was a void . So I set out on a vision -- how close in a weekend could I get to a minimum viable product that provided this value?

What it does

GenMetrix has a guided-tour that walks users through the process of uploading their genome for the first time. After which, information for the various traits/categories can be accessed from the sidebar menu entries for each. The overarching dashboard for a page contains three primary features:

  • A traits card page which presents the information in the most accessible and straightforward format possible
  • A data-table for more savvy users that condenses this trait information into a barebones and digestiable format
  • A chart for visualizing the weight/impact that a specific gene has , compared to others in the same category

How we built it

The backbone of the project revolves around a Node/JS utility I use to parse and serialize raw genome text files from the most common providers (23andMe, Ancestry, MyHeritage, and FamilyTree). It takes a text file the contents thereof, and does a clean mapping to a plain JSON object, creating something resembling a Dict structure where the keys are snp identifiers. Each keyed gene object provides the genotype for the given gene, plus some additional metadata.

The file upload is handled via HTML5 native FileReader API , using a standard HTML input element and creating a reader, then consuming the file with myRead.readAsText(input) .

The front-end is a plain HTML + JS + CSS templated landing page , with Vue.js, Vuex for state management, and a Vue admin template plus some addition utility libraries and components to build against rapidly.

In production or if the product were to do public access release, the genome utilities and serializer would be moved to the server-side , and I would use Node + Polka (much more performant than Express) to serve the static index.html of my built Vue frontend and landing page, with a single API endpoint as a handler for the genome parsing .

The parsing function would be offloaded to parallelized web/service workers to keep them off the main event thread, because it is fairly resource intense to parse and serialize a roughly ~16mb text file . I think with efficieny optimization tactics and distributed workers, the processing time could merge towards near-zero.

Challenges we ran into

Design inspiration was very difficult for me, so I needed to gather opinions on how best to present our analytics and data in an accessible and digestible manner to someone without context or background.

Accomplishments that we're proud of

I felt like given the time frame and ad-hoc collaboration, a lot of work got done. In my mind, this is a minimum viable product and completely demo-able. That in and of itself is a huge win, espcially given the timeframe and lack of manpower.

What we learned

You are much better off identifying people who can fill your weaknesses, to form a better group as a whole, faaaar in advance. Also, entire genome datasets being loaded in-browser will definitely brick your system if you are not picky about how you pass that data along (it needed to be sifted through and all riffraff discarded immediately in the parse callback, before it gets sent to the store as stateful data. Elsewise, bad newz bearz for your memory usage ;..;)
